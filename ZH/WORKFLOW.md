# WORKFLOW — 通用开发工作流规范

> 本文档定义 Phase 驱动开发的标准流程。适用于任何遵循"讨论→研究→计划→执行→审查→测试→完成"周期的项目。

***

## 零、三层架构

本项目采用 **Project → Milestone → Phase → Plan** 的层级结构：

| 层级            | 职责                     | 生命周期    | 文档位置                                      |
| ------------- | ---------------------- | ------- | ----------------------------------------- |
| **Project**   | 项目愿景、核心价值、架构约束         | 整个项目    | `.planning/PROJECT.md`                    |
| **Milestone** | 版本范围：该版本有哪些 Phase、哪些需求 | 一个版本周期  | `.planning/milestones/VN/`                |
| **Phase**     | 单个功能模块的讨论→实现→交付        | 数天到数周   | `.planning/milestones/VN/phases/0N-name/` |
| **Plan**      | Phase 内的最小执行单元（1-3 文件） | 数分钟到数小时 | `0N-PLAN-XX.md`（在 Phase 目录内）              |

**关键原则：**

- PROJECT.md 只有一页——项目愿景和约束，不含里程碑列表和需求清单
- 里程碑的 Phase 拆分和需求清单在 `.planning/milestones/VN/` 下单独管理
- 每次开始新里程碑时，通过子工作流 G 创建里程碑目录和文档

***

## 一、主工作流（10 步）

每次进入新 Phase 或继续当前 Phase 时，严格按以下顺序执行。

### 步骤检查表

| 检查条件                 | 未完成时做什么                          | 完成时做什么   |
| -------------------- | -------------------------------- | -------- |
| `0N-CONTEXT.md` 不存在  | **进入讨论模式**（步骤 1-2）               | → 下一步    |
| `0N-RESEARCH.md` 不存在 | **启动 PSR-Researcher 子智能体**（步骤 3） | → 下一步    |
| `0N-PLAN-*.md` 不存在   | **制定计划**（步骤 4）                   | → 下一步    |
| PLAN 未全部执行           | **执行下一个 PLAN**（步骤 5）             | → 下一步    |
| PLAN 全部执行完毕           | **启动 PSR-Reviewer → 创建/更新 `0N-REVIEW.md`**（步骤 6） | → 下一步    |
| REVIEW\.md 含 🔴 项    | **启动 PSR-Fixer 子智能体**（步骤 7）      | → 下一步    |
| UAT 未执行              | **创建 UAT.md → 通知用户测试**（步骤 8）     | → 下一步    |
| 用户反馈有问题              | **根据反馈修复**（步骤 9）→ 回到步骤 6         | → 下一步    |
| UAT 全部通过             | **确认完成**（步骤 10）                  | Phase 完成 |

***

### 步骤 1-2：讨论与清扫灰区

**触发：** CONTEXT.md 不存在。

**讨论粒度原则：** 逐系统、逐接口穷尽设计细节，讨论到**每条决策足够具体到可以直接编写代码**为止。不确定的数据字段、流程步骤、交互约定 → 追问到底，不留下模糊空间。

**主智能体执行：**

1. 读取 `.planning/PROJECT.md` 了解项目愿景和约束
2. 从 PROJECT.md 找到当前里程碑（如 V1），读取 `.planning/milestones/VN/STATE.md` 确认当前状态
3. 读取 `.planning/milestones/VN/ROADMAP.md` 确认 Phase 范围和依赖
4. 读取 `.planning/milestones/VN/REQUIREMENTS.md` 确认需求内容
5. 读取当前 Phase 依赖的所有已完成 Phase 的文档——**优先读 SUMMARY.md**，再按需读 CONTEXT.md 和 REVIEW\.md（了解已有决策和教训）
   - **特别关注：** 检查 SUMMARY.md 的"已知限制"中是否有指向本 Phase 的延后修复项——这些是之前 Phase 遗留给本 Phase 的待修复清单，必须纳入本 Phase 执行计划。
6. 读取项目的 CLAUDE.md 或等效架构约束文档
7. 读取 `.planning/research/PITFALLS.md`（如果存在）
8. 向用户提问，覆盖以下维度（每个维度追问到无灰色区域为止，逐条询问，可以的情况下调用ask工具）：
   - **功能边界：** 本 Phase 做什么、不做什么、输入输出是什么
   - **数据字段：** 每个数据类的字段名/类型/默认值/约束条件
   - **流程步骤：** 核心流程分几步，每步的前置条件、执行动作、后置结果
   - **接口约定：** 新增/修改的信号名称和签名、System 公开方法的参数和返回值
   - **交互约定：** 本 Phase 与已有系统的调用关系——调谁、怎么调、什么时候调
   - **数据格式：** JSON 配置文件的 schema、字段含义、示例值
   - **边界条件：** 什么情况下不触发、什么情况下报错、什么情况下降级处理
9. 逐条与用户确认需求，**当场分配 REQ-ID**
   （格式：`{DOMAIN}-NN`，领域前缀 SYS/WORLD/ITEM/BUILD/POP/PROD/PATH/ECON/UI）
10. 主动探索灰区 — 基于已讨论内容，推断用户可能遗漏的边界条件、异常情况、与其他系统的隐性交互，向用户提问确认
11. 创建 `0N-CONTEXT.md`（讨论结论，含架构决策编号 D-01\~D-N 和已分配 REQ-ID 的需求清单）
12. 创建 `0N-DISCUSSION-LOG.md`（讨论过程记录）
13. 向用户确认：**"不需要继续讨论本次 Phase 的话，就告诉我，开启步骤 3（研究）。"**
    - 用户如有补充 → 回到步骤 8 继续讨论
    - 用户确认完毕 → 讨论阶段结束，CONTEXT.md 和 DISCUSSION-LOG.md 已完成

**必须读取的文档：**

- `.planning/PROJECT.md`
- `.planning/milestones/{当前里程碑}/STATE.md`
- `.planning/milestones/{当前里程碑}/ROADMAP.md`
- `.planning/milestones/{当前里程碑}/REQUIREMENTS.md`
- `.planning/milestones/{当前里程碑}/RESEARCH.md`（如果里程碑研究已完成）
- 依赖 Phase 的 SUMMARY.md → CONTEXT.md → REVIEW\.md（按优先级）
- 项目的 CLAUDE.md
- `.planning/research/PITFALLS.md`（如果存在）

***

### 步骤 3：研究与探索

**触发：** CONTEXT.md 存在但 RESEARCH.md 不存在。

**执行者：PSR-Researcher 子智能体。**

PSR-Researcher 子智能体负责扫描代码库、分析影响范围、检查 PITFALLS、输出研究报告。主智能体调用它时只需说明当前 Phase 目录和 CONTEXT.md 路径。PSR-Researcher 将自己的发现写入 Phase 目录下的 `0N-RESEARCH.md`。

***

### 步骤 4：制定计划

**触发：** CONTEXT.md + RESEARCH.md 存在但 PLAN-\*.md 不存在。

**主智能体执行：**

1. 读取 `0N-CONTEXT.md`（讨论结论）
2. 读取 `0N-RESEARCH.md`（研究结论）
3. 拆分为 N 个 Wave（每个 Wave 有独立可验证的产出）
4. 每个 Wave 一个 PLAN 文件（`0N-PLAN-01.md`、`0N-PLAN-02.md`...）

**PLAN 拆分原则：**

- 数据定义先于系统逻辑（先建数据类，再建处理系统）
- 无依赖的 PLAN 可标注为无依赖（执行顺序任意，主智能体逐个串行执行）
- 每个 PLAN 产出不超过 3 个文件
- 如果 PLAN 涉及修改已有文件，在 PLAN 中标注"需确认影响范围"（引用步骤 3 RESEARCH.md 的"受影响文件"节；若 RESEARCH.md 未覆盖该文件，标注需步骤 5 补充研究）

***

### 步骤 5：执行计划

**触发：** PLAN 文件存在且未标注"已完成"。

**主智能体执行每个 PLAN 的规则：**

1. 读取 PLAN 文件
2. 如果 PLAN 涉及修改已有文件 → **先确认影响范围**：
   - 查阅步骤 3 的 RESEARCH.md "受影响文件"节：
     - **已覆盖：** 本 Phase 前置 PLAN 未改动该文件及其调用方 → 信任步骤 3 研究，直接执行
     - **未覆盖：** 或前置 PLAN 已改动了相关代码 → 启动 PSR-Researcher 做补充研究，确认安全后再改
3. 参考现有代码模板（搜索相邻文件，遵循已有模式）
4. 严格遵循项目架构约束
5. 完成后在 PLAN 文件末尾追加执行记录：

```markdown
## 执行记录
- 新增文件: xxx.gd（行数）
- 修改文件: xxx.gd（+N行，改动说明）
- 决策: （如有架构相关判断，记录在此）
- 未实现: （如有延后的内容，标注并说明原因）
```

***

### 步骤 6：审查

**触发：** 所有 PLAN 执行完毕。

**执行者：PSR-Reviewer 子智能体。**

PSR-Reviewer 子智能体负责对照架构约束检查所有新增和修改的代码，输出分级审查报告。主智能体调用它时只需说明 Phase 目录路径。PSR-Reviewer 将报告写入 Phase 目录下的 `0N-REVIEW.md`（不存在时创建，存在时覆盖），格式为 🔴/🟡/🟢 三级。

***

### 步骤 7：修复阻塞项

**触发：** REVIEW\.md 含 🔴 项。

**执行者：PSR-Fixer 子智能体。**

PSR-Fixer 子智能体负责读取 REVIEW\.md 中的 🔴 项，逐个修复违规代码，验证修复正确性，并将执行记录追加到 REVIEW\.md。

修复完成后，主智能体读取执行记录，确认修复内容无误。如有 ⚠️ 项（fixer 无法修复的项），主智能体评估后决定处理方式。

**循环规则：**

- PSR-Fixer 完成后，主智能体重新读取 REVIEW\.md
- 如果仍有 🔴 → 再次启动 PSR-Fixer（最多 3 轮）
- 3 轮后仍有 🔴 → 主智能体接管，人工修复
- 如果所有 🔴 变为 ⚠️ → 主智能体评估 ⚠️ 是否需要架构决策

***

### 步骤 8：用户人工测试

**触发：** REVIEW\.md 无 🔴 项。

**主智能体执行：**

1. 从所有 PLAN 文件中提取验证条件
2. 创建 `0N-UAT.md`（用户验收测试清单）
3. 明确告知用户测试项和预期结果
4. 等待用户反馈

***

### 步骤 9：修复错误

**触发：** 用户测试发现问题。

**主智能体执行：**

1. 根据用户反馈定位问题
2. 判断是否可在当前 Phase 修复：
   - **可以修复** → 简单修复（≤1 个文件）主智能体直接改 / 多文件修复启动 PSR-Fixer 子智能体 → 执行步骤 6（重新审查）
   - **不可修复**（依赖尚未实现的 Phase，或需要架构级决策）→ 记录延后修复：
     - 在 DISCUSSION-LOG.md 追加 "## 延后修复" 条目
     - 在 SUMMARY.md "已知限制" 中标注延后修复项（问题 + 来源 Phase + 目标 Phase + 优先级）
     - 如果是新发现的系统需求 → 追加到 `.planning/milestones/VN/REQUIREMENTS.md`（找到对应领域组，按 REQ-ID 格式追加，Traceability 表中 Phase 列指向目标 Phase）
     - 如果是重复出现的陷阱模式 → 追加到 .planning/research/PITFALLS.md
     - **继续执行**（不阻塞当前 Phase 的通过判定）

***

### 步骤 10：确认完成

**触发：** 用户测试全部通过。

**主智能体执行：**

1. 创建 `0N-SUMMARY.md`：Phase 概述、需求覆盖、交付内容、代码审查摘要
2. 更新 `.planning/milestones/{当前里程碑}/STATE.md`：Phase 状态改为 🟢 完成，指向下一个 Phase，追加决策记录
3. 更新 `.planning/milestones/{当前里程碑}/REQUIREMENTS.md`：
   - 将 CONTEXT.md 中已分配的 REQ-ID 抄录到 v1 Requirements 对应领域组（标记为 \[x]）
   - 在 Traceability 表追加本 Phase 的所有 REQ 行
4. 生成 Conventional Commits 格式的 commit message，**展示给用户确认**后提交到本地仓库

   **commit message 示例：**
   ```
   feat(phase-3): 居民与经济系统

   新增:
   - scripts/data/citizen_data.gd - 居民数据类
   - scripts/systems/citizen_system.gd - 居民管理系统

   修改:
   - scripts/systems/signal_bus.gd - 新增 citizen_spawned 等 3 个信号

   审查: 详见 .planning/milestones/V1/phases/03-citizen-economy/03-REVIEW.md
   ```

## 二、子工作流

### 子工作流 A：Phase 调整

**触发：** 用户需要对当前进行中 Phase 追加需求、重新讨论或拆分。

先判断用户意图，再执行对应分支。

#### 分支 1：追加需求

**触发：** 用户提出当前 Phase 的新需求。

1. 追加到 DISCUSSION-LOG.md
2. 判断归属：当前 Phase 还是新 Phase？
3. 如果是当前 Phase → 启动 PSR-Researcher 确认影响范围 → 创建新的 PLAN（编号 +1）→ 执行共同收尾 → 回归主工作流检查表，由检查表决定下一步
4. 如果是新 Phase → 执行共同收尾 → Phase 完成后处理
5. **绝不跳过研究阶段直接写代码**

#### 分支 2：重新讨论

**触发：** 用户说"重新讨论 Phase N"或认为当前讨论结论需要调整。

1. 读取现有 `0N-CONTEXT.md`、`0N-DISCUSSION-LOG.md`、`0N-RESEARCH.md`
2. 与用户重新讨论（遵循步骤 1-2 的讨论粒度原则）
3. 覆盖 `0N-CONTEXT.md`（新结论替代旧结论）
4. 追加 `0N-DISCUSSION-LOG.md`（标注 `## 重新讨论（YYYY-MM-DD）`）
5. 用 ask 工具询问用户：**"旧的研究报告和计划是否仍然有效？"**
   - 用户选"有效" → 执行共同收尾 → 回到主工作流检查表，由检查表决定下一步
   - 用户选"无效" → 删除 `0N-RESEARCH.md` + 所有 `0N-PLAN-*.md` + `0N-REVIEW.md`（如果存在）→ 执行共同收尾 → 回到主工作流检查表（检查表发现 RESEARCH 不存在 → 触发步骤 3）
6. **绝不跳过研究阶段直接写代码**

#### 分支 3：Phase 拆分

**触发：** 用户认为当前 Phase 范围过大，需要拆分为多个 Phase。

1. 与用户讨论拆分方案（基于灰区分析、依赖关系、复杂度评估）
2. 确认拆分方案后，执行以下操作：
   a. 创建新 Phase 目录（`0N-name/`）
   b. 创建新 Phase 的 CONTEXT.md 和 DISCUSSION-LOG.md（从原 Phase 迁移相关内容）
   c. 更新 ROADMAP.md：将原 Phase 替换为拆分后的多个 Phase
   d. 更新 STATE.md：更新 Phase 进度表 + 近期行动记录
   e. 更新 REQUIREMENTS.md：将原 Phase 的 REQ-ID 重新分配到新 Phase
   f. 确认内容已完整迁移后，删除原 Phase 目录
3. 执行共同收尾
4. 进入拆分后的第一个 Phase 讨论

#### 共同收尾（所有分支共享）

每个分支执行完毕后，检查以下文档是否需要更新：

| 文档 | 更新条件 |
|------|---------|
| DISCUSSION-LOG.md | 始终更新（记录讨论过程和决策） |
| STATE.md 近期行动 | 始终更新（记录本次操作） |
| ROADMAP.md | Phase 结构或依赖关系变化时 |
| REQUIREMENTS.md Traceability | REQ 分配变化时 |

***

### 子工作流 B：项目修改

**触发：** 用户需要修改已完成 Phase 的内容。

**流程：**

1. 读取目标 Phase 的全部 CONTEXT、PLAN、REVIEW、SUMMARY 文档
2. 讨论修改范围和影响
3. 修改相关文档（追加修订记录到 DISCUSSION-LOG.md）
4. 如需新增代码 → 启动 PSR-Researcher → 创建/修改 PLAN → 回归主工作流检查表
5. 如仅修改文档 → 主智能体直接更新
6. 完成后更新目标 Phase 的 SUMMARY.md（追加修订记录，标注修改日期和内容）

***

### 子工作流 C：框架审查

**触发：** 用户说"审查框架"或"检查架构健康度"。

**流程：**

1. 启动 PSR-Reviewer 子智能体（指定为框架审查模式）
2. PSR-Reviewer 审查整个项目输出报告到 `.planning/FRAMEWORK-REVIEW.md`
3. 主智能体读取报告，标记需要修复的项
4. 需要修复的转为计划追加（子工作流 A）
5. 值得记录的陷阱录入 .planning/research/PITFALLS.md

***

### 子工作流 D：里程碑收尾

**触发：** 用户说"里程碑完成"或所有 Phase 执行完毕。

**流程：**

1. 读取当前里程碑所有 Phase 的 SUMMARY.md，汇总为里程碑 `.planning/milestones/VN/SUMMARY.md`
2. 更新 `.planning/PROJECT.md`：当前代码库状态、已验证需求、已实现能力
3. 清理 STATE.md 历史记录，保留决策概要
4. 更新 codebase/ 下的静态快照文档（如果存在）
5. 生成里程碑摘要，展示给用户确认

***

### 子工作流 E：知识库更新

**触发：** 用户说"更新知识库"。

**流程：**

1. 扫描当前代码库目录结构
2. 对照 `.planning/codebase/` 目录中每个文档的主题，确认哪些需要更新
3. 更新过期的文档内容
4. 在文档顶部更新日期

**注意：** 日常开发中，PSR-Researcher 子智能体采用实时扫描策略，不依赖静态 codebase 快照。知识库更新仅作为人类可读的参考文档维护。

***

### 子工作流 F：项目初始化

**触发：** 用户开始一个新项目。

#### 入口判断

在开始提问前，先判断用户输入类型：

- **模式 A（文档摄入）：** 用户提供了已有设计文档或详细描述（如"这是我想做的游戏，设计文档如下……"）
- **模式 B（引导讨论）：** 用户只有一句话或模糊想法（如"我想做一个咖啡店模拟经营游戏"）
- **模式 C（项目重连）：** 用户已有 `.planning/` 目录和 PROJECT.md，需要重新审视项目方向

#### 模式 A：文档摄入

```
1. 读取用户提供的全部文档/描述
2. 按 PROJECT.md 节结构，从文档中提取信息填充每个节：
   - 设计概要 ← 从文档中提取游戏类型/机制/系统/玩法等完整内容
   - 项目概述 ← 从设计概要浓缩为一段 elevator pitch
   - 核心价值 ← 从文档中识别核心卖点
   - 对标作品 ← 如果文档提到了参考游戏
   - 约束 ← 如果文档提到了技术方向
3. 逐节向用户确认："我理解你的设计是这样的，对吗？"
4. 对文档没有覆盖的部分（如技术栈、架构风格、性能目标），
   进入模式 B 用引导提问补充
5. 汇总完整内容，写入 PROJECT.md
6. 创建目录结构和 CLAUDE.md
```

#### 模式 B：引导讨论

## 三、对话恢复流程

**触发：** 任何新对话开始时。

**主智能体执行：**

1. 读取 `.planning/PROJECT.md`，找到当前里程碑（如 V1）
2. 读取 `.planning/milestones/{当前里程碑}/STATE.md`
3. 找到当前 Phase 目录
4. 按修改时间排序列出所有文档
5. 找到最新文档 → 判断当前在哪个步骤
6. 向用户报告恢复状态：

```
📋 工作流恢复
里程碑: V1
Phase: N — {名称}（{状态}）
Phase 目录: .planning/milestones/V1/phases/0N-name/
最新文档: {最新文档路径}
判断: {当前步骤}
下一步: {下一步操作}
要继续吗？
```

**恢复判断规则：**

| 最新文档                  | 判断当前步骤                  |
| --------------------- | ----------------------- |
| 无任何 Phase 文档          | Phase 刚启动，进入步骤 1        |
| CONTEXT.md            | 讨论完成，进入步骤 3             |
| RESEARCH.md           | 研究完成，进入步骤 4             |
| PLAN 文件尾部有完整的执行记录     | 刚执行完某个 PLAN，继续执行或进入步骤 6 |
| REVIEW\.md 含 🔴 项    | 审查未通过，进入步骤 7         |
| REVIEW\.md 无 🔴 项    | 审查通过，进入步骤 8             |
| UAT.md                | 等待用户测试反馈                |

**重连询问：**

完成状态恢复、报告当前进度后，主动询问用户：

> 需要做项目级重连（重新审视项目方向）、里程碑级重连（调整当前里程碑范围）、或继续当前 Phase 的讨论吗？

- 用户说"重连项目" → 路由到子工作流 F 模式 C
- 用户说"重连里程碑"或"重连里程碑 VN" → 路由到子工作流 G 模式 C
- 用户说"继续讨论" → 路由到子工作流 A 分支 2（重新讨论）
- 用户说"继续" → 按恢复判断进入对应步骤

## 四、子智能体调用

本工作流依赖 3 个子智能体。调用时向它们说明当前 Phase 目录路径和具体任务即可。

| 子智能体               | 用途                                                                                                                    | 调用时机                            |
| ------------------ | --------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| **PSR-Researcher** | 扫描代码库、分析影响范围、输出研究报告。支持三级研究：项目级（PROJECT-RESEARCH.md）、里程碑级（.planning/milestones/VN/RESEARCH.md）、Phase 级（0N-RESEARCH.md） | 步骤 3、步骤 5（修改已有文件前）、子工作流 A/B/F/G |
| **PSR-Reviewer**   | 对照架构约束审查代码，输出分级报告。支持两种模式：常规审查（单个 Phase）和框架审查（全局）                                                                      | 步骤 6、子工作流 C                     |
| **PSR-Fixer**      | 读取 REVIEW\.md 的 🔴 项，逐个修复并验证，追加执行记录                                                                                   | 步骤 7、步骤 9（多文件修复）                |

### 子智能体选择规则

PSR 工作流中的"研究"和"搜索"是不同的任务，禁止用通用搜索替代 PSR 子智能体：

| 场景                     | 必须使用                    | 禁止使用   |
| ---------------------- | ----------------------- | ------ |
| 子工作流 F / 项目初始化 / 项目重连  | PSR-Researcher（项目级）     | search |
| 子工作流 G / 里程碑启动 / 里程碑重连 | PSR-Researcher（里程碑级）    | search |
| 步骤 3 / Phase 级研究       | PSR-Researcher（Phase 级） | search |
| 步骤 5 / 修改已有文件前确认影响     | PSR-Researcher          | search |
| 步骤 6 / Phase 审查        | PSR-Reviewer            | search |
| 步骤 7 / 修复阻塞项           | PSR-Fixer               | —      |
| 子工作流 C / 框架审查          | PSR-Reviewer（框架模式）      | search |
| 通用代码搜索（非 PSR 工作流场景）    | search                  | —      |

核心原则：**PSR 子智能体深度分析代码库架构和规划文档，通用 search 只做关键词匹配——两者不可互换。**

***

## 五、子智能体调用失败处理

1. 任何子智能体返回异常（无输出、输出不完整、明显错误）→ 重试 1 次
2. 第 2 次仍失败 → 主智能体接管该步骤
3. 记录到 DISCUSSION-LOG.md："## 子智能体故障：{名称} 返回异常，主智能体接管"

***

## 六、通用规则

1. **文档即状态机** — 文件存在=步骤完成，不存在=需要执行。具体情况以工作流各步骤的触发条件为准。
2. **所有注释使用中文**
3. **不确定时停止执行，向用户提问**
4. **禁止跳过研究阶段直接写代码**
5. **禁止修改代码时不先确认影响范围**
6. **所有 PLAN 执行完成后必须走审查 → 修复 → UAT 流程**
7. **子智能体输出以文件为准，不以返回消息为准** — 主智能体读取文件确认结果
8. **三层架构原则** — PROJECT.md 保持一页；里程碑文档在 .planning/milestones/VN/ 下；Phase 文档在 .planning/milestones/VN/phases/ 下
