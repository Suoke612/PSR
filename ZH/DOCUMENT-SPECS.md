# DOCUMENT-SPECS — 文档格式规范

> 本文档定义 `.planning/` 目录下所有文档的命名规则、节结构和必填字段。
> 创建或修改任何 `.planning/` 文档前，必须先读取本文档中对应的规范章节。

---

## 通用规则

### 文件命名概览

**项目级文档（放在 `.planning/` 根目录下）：**

| 文档 | 命名 | 位置 |
|------|------|------|
| 工作流规范 | `WORKFLOW.md` | `.planning/WORKFLOW.md` |
| 文档格式规范 | `DOCUMENT-SPECS.md` | `.planning/DOCUMENT-SPECS.md` |
| 项目描述 | `PROJECT.md` | `.planning/PROJECT.md` |
| 项目级研究 | `PROJECT-RESEARCH.md` | `.planning/PROJECT-RESEARCH.md`（可选，PSR-Researcher 项目级扫描输出） |
| 陷阱清单 | `PITFALLS.md` | `.planning/research/PITFALLS.md` |
| 框架审查 | `FRAMEWORK-REVIEW.md` | `.planning/FRAMEWORK-REVIEW.md` |

**里程碑级文档（放在 `.planning/milestones/VN/` 下，VN 为里程碑编号如 V1、V2）：**

| 文档 | 命名 | 示例 |
|------|------|------|
| 路线图 | `ROADMAP.md` | `.planning/milestones/V1/ROADMAP.md` |
| 需求清单 | `REQUIREMENTS.md` | `.planning/milestones/V1/REQUIREMENTS.md` |
| 状态追踪 | `STATE.md` | `.planning/milestones/V1/STATE.md` |
| 里程碑研究 | `RESEARCH.md` | `.planning/milestones/V1/RESEARCH.md` |
| 里程碑总结 | `SUMMARY.md` | `.planning/milestones/V1/SUMMARY.md` |

**Phase 级文档（放在 `.planning/milestones/VN/phases/0N-name/` 下，N 为 Phase 编号，XX 为 Plan 编号）：**

| 文档 | 命名 | 示例 |
|------|------|------|
| 讨论结论 | `0N-CONTEXT.md` | `03-CONTEXT.md` |
| 讨论日志 | `0N-DISCUSSION-LOG.md` | `03-DISCUSSION-LOG.md` |
| Phase 研究 | `0N-RESEARCH.md` | `03-RESEARCH.md` |
| 执行计划 | `0N-PLAN-XX.md` | `03-PLAN-01.md` |
| 代码审查 | `0N-REVIEW.md` | `03-REVIEW.md` |
| 用户测试 | `0N-UAT.md` | `03-UAT.md` |
| Phase 总结 | `0N-SUMMARY.md` | `03-SUMMARY.md` |

**参考文档（项目维护，工作流不强制读取）：**

| 文档 | 位置 |
|------|------|
| 架构设计 | `.planning/research/ARCHITECTURE.md` |
| 功能分析 | `.planning/research/FEATURES.md` |
| 技术栈 | `.planning/research/STACK.md` |
| 调研摘要 | `.planning/research/SUMMARY.md` |
| 代码库快照 | `.planning/codebase/ARCHITECTURE.md` 等 |

### 元数据头部

每个 Phase 专属文档以 YAML front matter 开头：

```yaml
---
phase: N
created: YYYY-MM-DDTHH:MM
updated: YYYY-MM-DDTHH:MM
---
```

### 内容规则

- 所有注释使用中文
- 引用其他文档或文件使用相对路径
- 决策编号格式：`D-01`、`D-02`...（每个 Phase 独立编号）

## PROJECT.md — 项目描述

**用途：** 记录项目的核心设计、约束条件和架构方向。是整个项目的"思想锚点"。任何子智能体从此文档获取项目全貌，无需用户重复描述。

**创建者：** 主智能体（子工作流 F）
**维护者：** 主智能体（子工作流 D 里程碑收尾时更新"当前代码库状态"节）
**读取者：** 主智能体（步骤 1、对话恢复）、PSR-Researcher 子智能体

**原则：** 篇幅自包含——不含里程碑列表、不含 Phase 拆分、不含需求清单。这些都在里程碑级别的文档中。不硬性限制行数，但以"新读者一次读完即理解项目全貌"为目标。设计概要可以详细，约束和决策保持精简。

**节结构：**

```markdown
# {项目名称}

## 项目概述
<!-- 一句话 + 一段话描述，项目的 elevator pitch -->

## 设计概要
<!-- ★ 核心主体节。项目完整的设计内容——
     游戏类型/主题/核心机制/核心循环/系统组成/
     玩法流程/世界观/角色/进度系统/数值方向……
     这是"这个项目具体做什么"的完整描述。
     长度不限，以说清楚为准。
     从用户的已有文档摄入，或通过引导讨论逐层构建。
     后续里程碑中的智能体从这里获取上下文。 -->

## 核心价值
<!-- 项目的独特卖点，为什么值得做 -->

## 对标作品与设计参考
<!-- 参考游戏列表，每项含借鉴要点 -->

## 约束
<!-- 表格：约束类型 / 内容 / 原因 -->

## 当前代码库状态
<!-- 简要描述当前已有系统、已实现能力，每个里程碑收尾时更新 -->

## 架构设计方向
<!-- 信号驱动 / ECS / JSON 配置等核心模式 -->

## 关键架构决策
<!-- 表格：决策 / 理由 / 状态。
     项目级的设计哲学选择，不随里程碑变化 -->

## 演进规则
<!-- 本文档的维护方式：何时更新、谁更新、更新什么 -->
```

**必填字段：** 项目概述、设计概要、约束、当前代码库状态。

## PROJECT-RESEARCH.md — 项目级研究报告

**用途：** 对整个项目代码库的全面扫描结果。可选，仅在子工作流 F 评估需要时创建。

**创建者：** PSR-Researcher 子智能体（子工作流 F — 项目级研究）
**读取者：** 主智能体（对话恢复）、里程碑级 PSR-Researcher

**生命周期：** 每个里程碑收尾时（子工作流 D）可选择性更新或覆盖。

**节结构：**

```markdown
# {项目名称} — Project Research

## TL;DR
<!-- 3-5 行摘要 -->

## 项目代码库全貌
<!-- 按目录列出所有模块及其职责 -->

## 已有基础设施
<!-- Autoload、配置系统、启动流程等 -->

## 架构合规性概述
<!-- 对照 CLAUDE.md 的架构约束做全局检查 -->

## 项目级风险
<!-- 跨里程碑的架构风险 -->
```

**必填字段：** TL;DR、项目代码库全貌。

## PITFALLS.md — 陷阱清单

**用途：** 记录项目开发中已知的致命陷阱、预防策略和对应 Phase。是 PSR-Researcher 和 PSR-Reviewer 的审查依据之一。

**位置：** `.planning/research/PITFALLS.md`

**创建者：** 主智能体（子工作流 F，创建空模板或初始清单）
**维护者：** 
- 任何 Phase 执行中发现新陷阱时，主智能体追加
- 子工作流 C（框架审查）后，经确认的新陷阱录入
**读取者：**
- PSR-Researcher 子智能体（步骤 3，扫描后逐项检查是否触发）
- PSR-Reviewer 子智能体（步骤 6，审查代码时对照检查）
- 主智能体（步骤 1 讨论，检查本 Phase 可能触发的陷阱）

**节结构：**

```markdown
# 陷阱清单 — {项目名称}

## 🔴 致命陷阱（P0 — 项目可能失败）
### N. {陷阱名称}
- 警告信号: {什么症状}
- 预防策略:
  - {具体措施}
  - ...
- 对应 Phase: {哪个 Phase 需要关注}

## 🟡 严重陷阱（P1 — 影响可玩性）
<!-- 同上 -->

## 🟢 中等陷阱（P2 — 影响开发效率）
<!-- 同上 -->

## 陷阱-Phase 映射表
<!-- 表格：陷阱 / P级 / 对应 Phase / 简要预防 -->
```

**必填字段：** 陷阱-Phase 映射表。

## FRAMEWORK-REVIEW.md — 框架审查报告

**用途：** 子工作流 C（框架审查）的输出，记录全局架构健康度。

**创建者：** PSR-Reviewer 子智能体（框架审查模式）
**读取者：** 主智能体（子工作流 C 完成后读取，转为计划追加）
**生命周期：** 每次框架审查覆盖上一次报告（同一文件，不累计）

**节结构：**

```markdown
# 框架审查报告 — {项目名称}

**审查日期:** YYYY-MM-DD

## 架构健康评分

| 维度 | 评分（1-10） | 问题数 | 简述 |
|------|------------|--------|------|
| 信号通信合规 | | | |
| 系统解耦度 | | | |
| 数据纯净性 | | | |
| 代码风格一致 | | | |
| 文档一致性 | | | |

## 🔴 架构层面问题
<!-- 逐条列出 -->

## 🟡 技术债务
<!-- 逐条列出 -->

## 改进路线图
<!-- 按优先级排列 -->
```

**必填字段：** 架构健康评分表、🔴 和 🟡 至少各有标题（可为空）。

## .planning/milestones/VN/REQUIREMENTS.md — 里程碑需求清单

**用途：** 该里程碑所有需求的单点记录，追踪实现状态。

**创建者：** 主智能体（子工作流 G — 创建轻量骨架）
**填充者：** 主智能体（步骤 10 — 每个 Phase 完成时填入该 Phase 的 REQ 条目和 Traceability）
**维护者：** 主智能体（子工作流 B 变更需求）
**读取者：** 主智能体（步骤 1、步骤 4）、PSR-Researcher 子智能体

**生命周期：** 子工作流 G 创建轻量骨架（只有里程碑设计概要 + 空白 Traceability 表），
每条 REQ 在对应 Phase 的步骤 10 完成时填入。本文档随里程碑推进逐渐丰满。

**节结构：**

```markdown
# Requirements — {里程碑名称}

## 里程碑设计概要
<!-- ★ 核心主体节。该里程碑实现的具体游戏内容。
     子工作流 G 创建时填入。——与 PROJECT.md 的设计概要对齐，但聚焦本里程碑范围。 -->

## v1 Requirements
### {领域} — {描述}
<!-- 每条 REQ 在对应 Phase 完成（步骤 10）时填入：
     - [x] **{DOMAIN}-NN**: {一句话描述}
     REQ-ID 格式：领域前缀（SYS/WORLD/ITEM/BUILD/POP/PROD/PATH/ECON/UI）
     + 两位数字编号，如 SYS-01、POP-03。
     步骤 10 同时更新 Traceability 表。
     子工作流 G 不展开 REQ——REQ 细节由主工作流步骤 1-2 逐 Phase 讨论决定。 -->

## V1 Out of Scope（本里程碑延后）
<!-- 里程碑讨论时确定的延后项。每条必须包含：
     - 需求名称
     - 延后到哪个版本
     - 延后原因 -->

## v2 Requirements（延后）
<!-- 已分配 REQ-ID 的后续版本需求 -->

## Out of Scope（永久排除）
<!-- 永久不做，附理由 -->

## Traceability
<!-- 表格：REQ-ID / 需求 / Phase / 状态。
     子工作流 G 创建时只有表头。
     每个 Phase 完成时由步骤 10 追加行。 -->
```

**必填字段：** 里程碑设计概要、Traceability 表头（子工作流 G 时）。
**重要：** V1 Out of Scope 中的每条延后项必须包含"延后原因"——不只是一个标题。
**REQ 填充时机：** 不在子工作流 G 中展开。每个 Phase 的步骤 10 完成时，将本 Phase 的 REQ 填入 v1 Requirements 并追加到 Traceability 表。

<!-- 关键架构决策汇总（跨 Phase 可见） -->

<!-- 已完成和待处理的操作列表 -->
