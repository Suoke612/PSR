# PSR Workflow — AI 辅助开发的文档驱动工作流

> **P**hase-driven development + **S**ub-agents + **R**ecovery = 防止 AI 在修水管时拆了你的房子。

---

## 这是什么？

PSR 是一套**文档驱动的状态机工作流**，专为 AI 辅助开发设计。它把开发过程拆成 10 个明确的步骤，用文件存在与否判断进度，用 3 个专用子智能体做质量把关。

核心思想很简单：**文件存在 = 步骤完成。** 主智能体每步检查目录，自己知道该干什么，不会迷路。

---

## 能解决什么问题？

| 痛点 | PSR 的解法 |
|------|-----------|
| 新对话开始，AI 忘了做到哪了 | 读取 STATE.md + 文档时间戳，自动恢复状态 |
| AI 不了解代码库就开始写代码 | 步骤 3 强制子智能体扫描代码库，输出研究报告 |
| 改新功能时破坏了已有系统 | 步骤 6 强制代码审查，步骤 7 自动修复架构违规 |
| 发现 bug 但依赖未实现的系统，修不了 | 步骤 9 延后修复机制——记录、继承、不阻塞 |
| 功能范围越做越大，失控 | 逐 Phase 讨论到每条决策可直接写代码的粒度 |
| 代码质量随迭代下降 | 每个 Phase 必经审查→修复→UAT 三道闸门 |
| 半路项目不知怎么接入 | 子工作流 F 模式 C（项目重连）直接整合 |
| 中断数月后回来，完全不记得思路 | 对话恢复流程 + 重连机制，从文档重建上下文 |

---

## 为什么选 PSR？

### 相比裸用 AI

- **状态不丢：** 文档存在磁盘上，对话丢失也能恢复
- **质量不退化：** 每个 Phase 强制审查和修复，架构违规不会累积
- **上下文不炸：** 3 层架构分离关注点，CLAUDE.md 保持精简

### 相比传统项目管理系统

- **零配置：** 只是一堆 Markdown 文件，不需要安装任何工具
- **AI 原生：** 子智能体会自动读取 CLAUDE.md 了解项目规则，更换项目无需重配子智能体
- **单人优化：** 没有多人协作的 ceremony，每一步就是下一件事

---

## 适合谁？

- **独立游戏开发者：** 一个人+AI 做项目，需要结构但不想要流程负担
- **独立软件开发者：** 适用于任何单人主导的软件项目，不限于游戏
- **AI 辅助编程的探索者：** 想系统化地用 AI 做开发，而不是"写完再说"
- **需要长期维护项目的开发者：** 项目可能中断几周/几月再回来，需要能重建上下文

**不适合：** 大型多人团队（需要更重的工作流）、一次性脚本任务。

## 核心架构

### 三层文档结构

```
Project (.planning/PROJECT.md)     ← 项目愿景、设计概要、约束（一页）
  └─ Milestone (milestones/VN/)     ← 版本范围、Phase 拆分、需求清单
       └─ Phase (phases/0N-name/)   ← 单个功能模块的讨论→交付（10 步）
            └─ Plan (0N-PLAN-XX.md) ← 最小执行单元（1-3 个文件）
```

- **Project** 说清楚"做什么、有什么约束、用什么技术"
- **Milestone** 说清楚"这个版本包含哪些 Phase、有哪些需求"
- **Phase** 是真正干活的地方——讨论、研究、计划、执行、审查、测试
- **Plan** 是最小执行单元，每个产出不超过 3 个文件

### 三个子智能体

| 子智能体 | 做什么 | 权限 |
|---------|------|------|
| **PSR-Researcher** | 扫描代码库、分析影响范围、检查陷阱 | 只读代码，写 RESEARCH.md |
| **PSR-Reviewer** | 对照架构约束审查代码，输出分级报告 | 只读代码，写 REVIEW.md |
| **PSR-Fixer** | 读取 🔴 项逐个修复，最多 3 轮 | 可修改代码 |

所有子智能体的审查标准来自 `CLAUDE.md`——换项目只需改一个文件，子智能体无需重配。

## 核心工作流（10 步）

```
步骤 1-2: 讨论与清扫灰区  →  创建 CONTEXT.md + DISCUSSION-LOG.md
步骤 3:   研究与探索       →  PSR-Researcher 输出 RESEARCH.md
步骤 4:   制定计划         →  主智能体创建 PLAN-*.md
步骤 5:   执行计划         →  主智能体逐个实现 PLAN
步骤 6:   代码审查         →  PSR-Reviewer 输出 REVIEW.md
步骤 7:   修复阻塞项       →  PSR-Fixer 消除 🔴 项（最多 3 轮）
步骤 8:   用户验收测试     →  创建 UAT.md → 用户人工测试
步骤 9:   修复错误         →  根据反馈修复 / 不可修复→延后
步骤 10:  确认完成         →  创建 SUMMARY.md → 更新 STATE.md → commit
```

主智能体通过**检查表**自动判断该执行哪一步——文件不存在就执行，存在就跳过。不需要记住步骤。

## 命令速查

### 日常开发

| 你说什么 | 触发什么 |
|---------|---------|
| `开始 Phase N` | 进入步骤 1-2 讨论 |
| （Phase 执行中自动推进） | 步骤 3→4→5→6→7→8 自动流转 |
| `追加需求：{描述}` | 子工作流 A 分支 1 — 当前 Phase 新增需求 |
| `重新讨论 Phase N` | 子工作流 A 分支 2 — 重新讨论当前 Phase 结论 |
| `修改 Phase N：{描述}` | 子工作流 B — 修改已完成 Phase |

### 项目管理

| 你说什么 | 触发什么 |
|---------|---------|
| `初始化 PSR 工作流` 或 `开始新项目` | 子工作流 F — 创建项目骨架 |
| `开始里程碑 V1` | 子工作流 G — 创建里程碑文档 |
| `里程碑完成` | 子工作流 D — 收尾当前里程碑 |
| `审查框架` | 子工作流 C — 全局架构健康度检查 |
| `更新知识库` | 子工作流 E — 更新 codebase/ 文档 |

### 对话恢复与重连

| 你说什么 | 触发什么 |
|---------|---------|
| `恢复工作流` 或新对话开始 | 恢复流程 — 自动判断当前步骤 |
| `重连项目` | 子工作流 F 模式 C — 重新审视项目方向 |
| `重连里程碑` 或 `重连里程碑 V1` | 子工作流 G 模式 C — 调整里程碑范围 |

### 重连场景速查

| 你遇到的情况 | 用这个命令 |
|------------|----------|
| 项目方向变了（比如从 RPG 改模拟经营） | `重连项目` |
| 里程碑范围太大/太小，需要调整 Phase 拆分 | `重连里程碑` |
| 停了好几个月回来，忘了做到哪 | `恢复工作流`（自动） |
| 需求变化导致 Phase 计划无效 | `重连里程碑` |
| 技术选型变化，架构约束需要更新 | `重连项目` |

## 接入指南

### 已有项目想接入 PSR？只需 2 步：

**步骤 1：** 将 `.planning/` 目录复制到你的项目根目录。

**步骤 2：** 对 AI 说：

> 读取 `.planning/WORKFLOW.md`，帮我初始化 PSR 工作流

Agent 会：
- 读取工作流规范 → 执行子工作流 F
- 引导你完成项目骨架（PROJECT.md + CLAUDE.md + 全部目录）
- 后续每次新对话，Agent 从 CLAUDE.md 自动恢复 PSR 工作流状态

### 创建子智能体（可选）

在 Trae IDE（或其他支持子智能体的编辑器）中创建 3 个子智能体：
- **PSR-Researcher** — 研究员（参照 `PSRWORKFLOW/ZH/PSR研究.md`）
- **PSR-Reviewer** — 审查员（参照 `PSRWORKFLOW/ZH/PSR审核.md`）
- **PSR-Fixer** — 修理工（参照 `PSRWORKFLOW/ZH/PSR修理.md`）

不创建子智能体也能正常使用工作流——主智能体会在需要时提示你手动接管对应步骤。

## 目录结构总览

```
.planning/
├── WORKFLOW.md                  # ★ 工作流规范（本工作流的核心定义）
├── DOCUMENT-SPECS.md            # ★ 文档格式规范（每类文档的结构和必填字段）
├── PROJECT.md                   # 项目愿景、设计概要、约束、架构方向
├── PROJECT-RESEARCH.md          # 项目级代码库研究（可选）
├── FRAMEWORK-REVIEW.md          # 框架审查报告
├── research/
│   └── PITFALLS.md              # 陷阱清单（已知致命问题 + 预防策略）
├── codebase/                    # 代码库分析快照（人类参考，非强制）
└── milestones/                  # 里程碑级文档
    └── V1/
        ├── ROADMAP.md           # Phase 拆分 + 依赖关系 + 成功标准
        ├── REQUIREMENTS.md      # 需求清单 + 可追溯性表
        ├── STATE.md             # 状态追踪（当前 Phase、进度、决策记录）
        ├── RESEARCH.md          # 里程碑级宏观影响分析
        ├── SUMMARY.md           # 里程碑总结（收尾时创建）
        └── phases/
            └── 01-name/
                ├── 01-CONTEXT.md        # 讨论结论 + 架构决策
                ├── 01-DISCUSSION-LOG.md # 讨论过程记录
                ├── 01-RESEARCH.md       # Phase 研究报告
                ├── 01-PLAN-01.md        # 执行计划（Wave 1）
                ├── 01-REVIEW.md         # 代码审查报告
                ├── 01-UAT.md            # 用户验收测试清单
                └── 01-SUMMARY.md        # Phase 执行总结
```

## 注意事项

1. **CLAUDE.md 是唯一规则源** — 子智能体的行为标准都从这里读取，换项目改一个文件就行
2. **不跳过研究** — 步骤 3 强制扫描代码库，防止不了解现状就写代码
3. **不跳过审查** — 步骤 6 强制对照架构约束检查，违规不会累积
4. **延后修复不丢** — 修不了的 bug 会记录到 SUMMARY.md，后续 Phase 在步骤 1 时自动继承
5. **文档即状态** — 子智能体以文件输出为准，不以回复消息为准
6. **.planning/ 放入 .gitignore** — 规划文档是本地的，不随代码提交（或自行决定）
